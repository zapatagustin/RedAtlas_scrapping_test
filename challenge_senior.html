Contexto

Red Atlas opera scrapers de producción sobre portales inmobiliarios en Puerto Rico, Colombia, Argentina y España. Este challenge simula exactamente eso: extraer datos reales de un portal real que resiste el scraping.

No buscamos código perfecto. Buscamos criterio técnico, decisiones fundamentadas y evidencia de haber lidiado con sistemas que no quieren ser scrapeados.

Objetivo

Construir un scraper que extraiga listados de propiedades en Puerto Rico desde Zillow (zillow.com/pr/), aplicando los siguientes filtros:

  • Tipo de propiedad: Houses y Apartments
  • Estado del listing: For Sale y For Rent
  • Mínimo 2 bedrooms

Los datos deben almacenarse en SQLite con el schema definido más adelante. El scraper debe avanzar hasta donde la IP lo permita sin uso de proxies — no se penaliza llegar a un límite de bloqueo.

Zillow requiere IP de Estados Unidos. Si estás fuera de EEUU, necesitás conectarte a un servidor US antes de ejecutar el scraper. ProtonVPN y Windscribe tienen tier gratuito con servidores en EEUU — sin tarjeta de crédito.

Schema SQLite esperado

CREATE TABLE listings (
    zpid           TEXT PRIMARY KEY,
    url            TEXT,
    listing_status TEXT,
    property_type  TEXT,
    latitude       REAL,
    longitude      REAL,
    price          INTEGER,
    bedrooms       INTEGER,
    bathrooms      REAL,
    living_area    INTEGER,
    address        TEXT,
    description    TEXT,
    photo_url      TEXT,
    data           TEXT,       -- JSON completo de la propiedad
    scraped_at     TEXT,
    status         TEXT DEFAULT 'pending'
);

Pistas técnicas

No es obligatorio usarlas, pero ayudan a arrancar sin perder tiempo en lo que no se evalúa:

  • Zillow embebe todos los datos de cada página en un <script id="__NEXT_DATA__"> dentro del HTML — si usás browser, no hace falta parsear selectores. Ese JSON tiene todo: zpid, precio, coordenadas, fotos, descripción.
  • Para identificar los endpoints internos, el tab Network → Fetch/XHR en DevTools es el punto de partida. Navegá a zillow.com/pr/ con los filtros aplicados y observá las llamadas que hace la página.
  • El sistema anti-bot de Zillow es PerimeterX. Para requests directas sin browser, el fingerprinting TLS es el obstáculo principal.
  • La URL de búsqueda con filtros ya aplicados tiene esta forma: zillow.com/pr/?searchQueryState=... — el parámetro es un JSON URLencodeado que incluye el estado del mapa, filtros de tipo de propiedad y número de bedrooms.

Diferenciadores

El scraping por DOM con browser es un approach completamente válido y es lo que se espera como solución base. Las siguientes son formas de destacar, no requisitos:

  • ★API interna sobre DOM. Si se usa browser, interceptar o replicar las llamadas XHR/Fetch internas para obtener los datos como JSON directamente — en lugar de extraerlos del HTML o del __NEXT_DATA__. Más robusto ante cambios de UI.
  • ★Extracción sin navegador. Replicar las llamadas HTTP directamente con requests, httpx, curl_cffi u otra librería sin abrir ningún browser. Requiere manejar la protección PerimeterX a nivel de TLS/headers — difícil, pero posible.

Requisitos transversales — aplican a todos los tiers

  • Resume/pause: si el proceso se interrumpe, retoma desde donde quedó sin volver a procesar listings ya guardados. Usar el campo status en SQLite.
  • Reintentos con backoff: ante error HTTP o timeout, reintentar con espera exponencial. Máximo de reintentos configurable.
  • Logging: cada página procesada, cada error, cada reintento. Nivel de detalle básico: INFO / WARN / ERROR.
  • No duplicados: usar zpid como clave única. Si un listing ya existe en DB con status = done, no reprocesar.
  • README con instrucciones de instalación y ejecución en una sola máquina.

Sin proxies. No se proveen proxies para este challenge. El scraper debe operar desde una sola IP y avanzar hasta donde Zillow lo permita. No se penaliza llegar a un límite de bloqueo — sí se penaliza no documentarlo. Ver sección de diferenciadores para la extensión opcional de proxy.

Preguntas de arquitectura

Responder en el README o en un archivo separado. No hay respuesta única correcta — buscamos entender cómo razonás ante problemas reales de producción. Respuestas de 3 a 10 líneas por pregunta.

  1. Tu scraper con Playwright lleva 2 horas corriendo, no hay errores, pero el proceso está consumiendo 1.8 GB de RAM y cada página tarda el doble que al inicio. ¿Cómo diagnosticás qué está pasando y cómo lo resolverías? ¿Qué cambiarías en tu código o en la config de PM2?
  2. El scraper corrió sin errores durante 3 días, pero al revisar la base de datos te das cuenta que la mitad de los registros tienen price = null y address = null. No hubo ningún crash ni log de error. ¿Qué pudo haber pasado y cómo diseñarías el sistema para detectar esto antes de que ocurra?
  3. Tenés un scraper corriendo 24/7 en PM2 y necesitás deployar un fix urgente. El proceso está en el medio de una corrida, procesando la página 15 de 40. ¿Cómo hacés el deploy sin perder el progreso ni duplicar registros ya procesados?
  4. Necesitás procesar 500 URLs en paralelo en Node.js pero con un máximo de 10 concurrentes. ¿Cómo lo implementás? Si uno de los 10 workers falla y hace retry con backoff de 30 segundos, ¿qué problema podría aparecer cuando todos los workers fallan al mismo tiempo? ¿Cómo lo evitás?
  5. Dado el scraper que construiste, ¿cómo lo modificarías para procesar miles de listings en paralelo sin que Zillow te bloquee? ¿Qué partes del código cambiarían, qué limitaciones tiene tu approach actual y cómo las resolverías?
  6. ¿Cómo integrarías un pool de proxies rotativos a tu scraper? Describí los cambios concretos en tu código — cómo elegirías qué proxy usar en cada request, cómo detectarías que un proxy fue bloqueado y cómo lo sacarías de rotación.
